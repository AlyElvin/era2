# PRD — Очередь генераций (Generation Queue)

## Problem Statement

ERA2 — агрегатор нейросетей: одна подписка даёт доступ к 90+ моделям. Задачи на генерацию выполняются не мгновенно и обрабатываются очередью с лимитом одновременных задач. Сейчас в продукте нет экрана, где пользователь видит свою очередь и управляет ей. Пользователю некуда смотреть «что сейчас генерится, что в очереди, что упало», и нет глобального индикатора прогресса при уходе с экрана.

## Solution

Новый экран **«Очередь генераций»** (`/queue`), который «живёт во времени»: задачи сами продвигаются по статусам через клиентский мок-движок (бэкенда нет). Пользователь видит сводку-счётчики, фильтрует/сортирует/ищет задачи и управляет ими (отмена / повтор / скачать-заглушка / удалить). Плюс **глобальный плавающий статус-бар генераций**, видимый поверх любого экрана, пока есть активные задачи, читающий тот же единый стор.

## User Stories

1. Как пользователь, я вижу список своих задач генерации со статусами и живым прогрессом, чтобы понимать состояние очереди.
2. Как пользователь, я вижу 4 реактивных счётчика (В очереди / Идёт / Готово / Ошибка), чтобы быстро оценить картину.
3. Как пользователь, я фильтрую по статусу, сортирую (новые/старые) и ищу по тексту промпта, чтобы найти нужную задачу.
4. Как пользователь, я отменяю/повторяю/удаляю задачи и скачиваю готовые (заглушка), чтобы управлять очередью.
5. Как пользователь, я нажимаю «Очистить готовые» и могу отменить действие (undo), чтобы не потерять нужное по ошибке.
6. Как пользователь, я вижу осмысленные состояния пустоты, загрузки и ошибки инициализации.
7. Как пользователь, я вижу глобальный индикатор генераций поверх любого экрана и по клику попадаю на `/queue`.
8. Как пользователь, состояние моей очереди сохраняется между перезагрузками страницы.

## Implementation Decisions

- **Стек/архитектура:** React 19 + TS strict, Tailwind v4, FSD. Следуем структуре файлов из `тз.md` §1.
- **State management:** React Context + `useReducer` (`QueueProvider` + `useQueue`). Redux не используется (в проекте его нет; `тз` предписывает useReducer). **Единый глобальный провайдер** монтируется на уровне `app` (`AppProviders`), чтобы и страница `/queue`, и глобальный статус-бар читали один стор — счётчики и прогресс всегда совпадают.
- **Движок очереди:** `queueEngine.ts` — тики через `setInterval`/`setTimeout`, `MAX_CONCURRENT = 2`, FIFO по `createdAt`, рост `progress`, ~15% случайных сбоев, разные длительности по типу (video/audio дольше text/image), корректная чистка интервалов, немедленная остановка при `cancel`. Единый источник правды — стор (диспатч экшенов), движок не мутирует state напрямую.
- **Конечный автомат:** `queueReducer.ts` — переходы `queued → running → done | failed`, `running/queued → canceled` (cancel), `failed/canceled → queued` (retry), `done → (removed)` (clear done / delete).
- **Маршрутизация:** проект использует собственный роутер `@/shared/routing` (не react-router). Добавляем маршрут `/queue` в `app/router` и ссылку в навигацию. Клик по статус-бару → `navigate("/queue")`.
- **Персистентность:** `localStorage` (ключ `era2:generation-queue`). При восстановлении задачи в статусе `running` переводятся в `queued` (re-queue) — чистое поведение без устаревшего прогресса; задокументировать в README.
- **Инициализация:** эмуляция задержки загрузки ~600 мс (LoadingState) и возможного сбоя инициализации (ErrorState с «Повторить»).
- **Дизайн-токены:** только токены дизайн-системы из `src/app/styles/styles.css` (semantic + `era` raw). Акцент `#E85420`, тёмная тема «warm coal» (canon). Шрифт Geist / Geist Mono (уже подключены). Статусы: queued — нейтральный, running — оранжевый (primary), done — зелёный, failed — красный (destructive), canceled — приглушённый.
- **UI-kit:** переиспользуем существующие `shared/ui` (shadcn: button, card, input, dropdown-menu, progress, alert-dialog, tooltip, sonner) и `shared/ui/era` (Chip, Placeholder). Свои примитивы только если нужного нет.
- **Бонусы в scope:** Undo на delete/clear-done (через `sonner`); a11y (focus, aria, клавиатура, `prefers-reduced-motion`); анимации появления/удаления через `framer-motion` (установлен).
- **Адаптив:** Desktop (≥1024) — строки (`TaskRow`); Mobile (≤480) — карточки (`TaskCard`) в стек, статистика 2×2, чипы со скроллом по горизонтали; планшет не ломается. `useIsMobile` уже есть (брейкпоинт 768) — при необходимости добавить хук/медиазапросы под нужные точки.

## Testing Decisions

- Нет настроенного тест-раннера; юнит-тесты — вне scope (по решению пользователя). Верификация: `yarn lint`, `yarn tsc --noEmit`, `yarn build`, и ручная проверка через `yarn dev` (http://localhost:8080).
- Каждый слайс должен собираться (build green) и проходить lint + type-check.

## Out of Scope

- Реальный бэкенд, сетевые вызовы, авторизация.
- Остальные страницы продукта; **верстать экран чата не нужно** (только глобальный индикатор поверх простого фона).
- Юнит-тесты / Vitest, фильтр по типу, отдельная работа над светлой темой, виртуализация списка, drag-to-reorder (бонусы, не выбраны).
- Пиксель-перфект до единицы.
